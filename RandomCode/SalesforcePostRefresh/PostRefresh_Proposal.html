The problem

Sandbox refreshes are high-risk,
low-visibility events

Every refresh lands production data, live email addresses, and active integrations in a non-production environment. Without a structured process, things go wrong.

✉️

Emails sent to real users

Sandbox workflows fire against real production email addresses until deliverability is disabled.

🔐

PII exposed to developers

Contact and Lead records carry real names, phone numbers, and payment tokens until masking runs.

🔗

Integrations pointing at production

Named credentials still point at production endpoints. A sandbox test triggers real middleware.

📋

No consistent checklist

Steps live in tribal knowledge. Different engineers follow different sequences and miss different steps.

⏱️

Hours of manual work

An experienced engineer takes 3–5 hours. A new joiner takes a full day and still misses things.

🚫

No audit trail

When something goes wrong there's no log of what ran, in what order, or whether it succeeded.

The solution

A metadata-driven orchestration platform

Layer 1
Custom Metadata Types
Activity registry — sequence order, type, class name, dependencies. Version-controlled. Zero code to add activities.
Layer 2
Apex Orchestrator
Reads registry, resolves dependencies, dispatches each activity to the correct mechanism, chains batch jobs via finish().
Layer 3
LWC Dashboard
Live status polling, phase grouping, individual triggers, manual sign-off buttons, exportable execution log.

Activity classification

Four execution types, one orchestrator

Batch

Database.Batchable — large record volumes

Use when the activity processes more records than a single transaction can handle. Chains via finish().

  • Mask PII on Contact, Lead, custom objects
  • Invalidate user email addresses (.invalid)
  • Delete email message and audit log records
  • Wipe payment tokens across account records
Metadata

IPostRefreshHandler — single transaction

Use when the activity changes org configuration and completes in milliseconds. Runs synchronously.

  • Disable email deliverability
  • Update custom settings (env config flags)
  • Activate or deactivate flows by env tag
  • Set debug log levels for active developers
API / Queueable

Queueable + callout — external systems

Use when the activity requires an HTTP callout. Queueable allows callouts; batch context does not.

  • Revoke OAuth tokens on the auth server
  • Ping integration endpoints for health check
  • Post Slack / Teams refresh notification
  • Create a Jira compliance ticket
Manual

Human gate — cannot be automated

Orchestrator pauses and marks as "Awaiting Action". Developer completes and clicks Mark Complete.

  • Re-authenticate named credentials (OAuth)
  • Re-enter API keys unavailable from production
  • Validate third-party integration handshake
  • Install or upgrade managed packages
Classification decision tree

Four questions that classify any activity

Q1
Does this activity require a human to log in, make a judgment call, or complete an action in an external UI that has no API?
Yes
Create a Manual task. Document exactly what the human must do and what "done" looks like.
No — continue ↓
This activity can be automated. Move to Q2.
Q2
Does this activity need an HTTP callout to an external system — auth server, middleware, Slack, Jira, or SSO provider?
Yes
Build a Queueable implementing Queueable + AllowsCallouts. Read endpoints from Named Credentials.
No — continue ↓
No callout needed. Move to Q3.
Q3
Does this activity need to process, update, or delete more than a few hundred records — i.e. would it hit governor limits in one transaction?
Yes
Build a Batch extending PostRefreshBatchBase. Implement Database.Stateful to carry chain state into finish().
No — continue ↓
Single-transaction operation. Move to Q4.
Q4
Is this a pure org config change — custom setting, flow activation, email deliverability, debug log, or static resource?
Yes
Build a Metadata handler implementing IPostRefreshHandler. Runs synchronously — orchestrator chains immediately after.
No
Revisit — return to Q1. If still unclear, raise as a spike with the tech lead before writing any code.

💡
The classification is not permanent
A Manual task today may become a Technical Story next quarter as Salesforce expands its API surface. Build a quarterly review into planning so Manual tasks are regularly re-evaluated.
