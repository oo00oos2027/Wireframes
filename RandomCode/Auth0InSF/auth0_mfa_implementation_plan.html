Overview
What we are implementing
Multi-factor authentication for all Salesforce Experience Cloud community users, powered by Auth0 as the identity provider. Users will authenticate through Auth0 instead of Salesforce's native login, with MFA enforced at every login.
🔐

Auth0 as identity provider

Auth0 replaces Salesforce's native login for all community users. SAML 2.0 SSO connects Auth0 to Salesforce Experience Cloud.

📱

MFA for every login

Every community user must enrol in at least one MFA factor: SMS, Email OTP, TOTP (authenticator app), or Push notification.

🔄

JIT user provisioning

Salesforce users are created and updated automatically via a Just-in-Time handler when a user successfully authenticates through Auth0.

📊

User data migration

All existing users are classified by data quality (Scenario A–D) before being imported into Auth0. Phone and email flags drive MFA factor eligibility.

🛟

Admin recovery tooling

A role-restricted MFA reset tool in Salesforce allows authorised support staff to reset a user's MFA with a full audit trail.

🚦

Two user creation paths

New users can self-register via the community portal, or be created directly by an admin in Salesforce. Both paths produce a linked Auth0 account.

User creation
How new users get an Auth0 account after go-live
The bulk import covers existing users only. For every new user created after go-live, Auth0 must receive their data in real time through one of two flows.

The bulk import was a one-time event. It ran at migration time against your existing user population. After go-live, Auth0 never receives phone or email data passively — your Salesforce code must actively push it to Auth0 the moment a new user is created. The mechanism differs depending on whether the user self-registers or an admin creates them directly.

Key point on phone data: Auth0 does not have a native phone field on its user object. Phone is stored in user_metadata.phone and must be pushed explicitly via the API. The phone uniqueness check — which was a batch SQL query during migration — must now happen in real time in Apex at the moment the phone number is entered.

A

Self-registration via community portal

User drives this. Auth0 account created at signup.
1

User visits Experience Cloud registration page

Fills in name, email, password. Phone is optional on the form but prompted for MFA factor eligibility.

2

Real-time checks run on form submit (Apex)

Email unique in Auth0? Phone unique across all users? Phone normalised to E.164? Email format valid? If any check fails, form shows an error — nothing is written yet.

3

Apex calls Auth0 signup API

Calls POST /dbconnections/signup with email and password. Phone is included in user_metadata.phone. If phone is already in system → phone_shared: true flag set immediately.

4

Auth0 creates user account + returns user_id

Auth0 enforces email uniqueness natively — will reject a duplicate. MFA scenario flag (mfa_scenario: "A" or "C" etc.) written to user_metadata.

5

Apex creates Salesforce Contact + Community User

Registration handler creates the SF records. Auth0_User_ID__c is populated immediately from the Auth0 response.

6

User logs in for the first time

Auth0 → SAML assertion → JIT handler updates the SF user record with MFA metadata. Post-Login Action detects no MFA enrolled.

7

MFA enrollment triggered

Factor picker shows options based on mfa_scenario flag already in user_metadata. If phone was provided and is unique, SMS is available.

B

Admin creates user directly in Salesforce

Admin drives this. Auth0 account created by a trigger after SF insert.
1

Admin opens Setup or a custom user creation form

Fills in name, email, phone, Profile, and community license. This is a standard Salesforce user creation — nothing Auth0-specific yet.

2

Real-time checks run before insert (Apex Trigger / validation)

Email unique in SF? Phone unique across all SF users? Phone normalised to E.164? These replace the batch queries from migration time. Must be synchronous — block insert if checks fail.

3

Salesforce User record is created

Phone, email, profile, and license saved in Salesforce. At this point, Auth0 still has no knowledge of this user.

4

Apex Trigger fires after insert → calls Auth0 Management API

This is the key difference from Flow A. The trigger calls POST /api/v2/users with email, phone (in user_metadata), and MFA scenario flag. This creates the Auth0 account in real time. If the trigger fails, the Sc 44 safety net kicks in.

5

Auth0 returns user_id → written back to SF

Trigger writes the Auth0 user_id to Auth0_User_ID__c on the SF User record in a second DML update. Both systems are now linked.

6

SF emails the user with a login link

Standard community welcome email. User clicks the link and is redirected to the Auth0 Universal Login page — not Salesforce's login.

7

MFA enrollment triggered on first login

JIT updates the SF user from SAML attributes. Post-Login Action triggers enrollment. Factor options based on phone data already in user_metadata from the trigger.

Key decisions and technical notes

Where does phone go in Auth0?
Auth0 has no native phone field. Phone is stored in user_metadata.phone and must be explicitly sent in every API call that creates or updates the user.
When does the phone uniqueness check happen?
During migration: a batch SQL query. After go-live: synchronous Apex validation at form submit (Flow A) or before trigger executes (Flow B). It must block the insert if the phone is already in use.
What if the Apex trigger fails in Flow B?
The SF user is created but Auth0 has no account. The Sc 44 safety net — a daily report of Auth0_User_ID__c = null users plus a one-click creation button — catches and resolves these gaps.
Who creates the SF user in Flow A?
Auth0 creates the Auth0 account first. Apex then creates the SF Contact and Community User in the same registration handler call. The Auth0 user_id is stored immediately — JIT only updates the record on first login, it does not create it.
What about email uniqueness?
Auth0 enforces this natively — it rejects a signup or Management API call with a duplicate email. In Flow A this surfaces as a registration error. In Flow B the trigger must handle the Auth0 API error and surface it to the admin.
What MFA factors are available for new users?
Determined at the point of account creation based on phone status. Phone provided and unique → Scenario A (all factors). Phone shared → Scenario B (no SMS). No phone → Scenario C (no SMS). The Post-Login Action reads the scenario flag from user_metadata.
